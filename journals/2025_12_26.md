		- 11:50 梳理 Logseq 工作流 (JDD + PARA) - 核心原则 (Principles) [[KM]]
			- 流 (Stream)：所有输入都在 Journal，不要直接去页面写内容。
				- 根据以往的整理个人笔记的经验来看，如果不把入口限制在一个地方，会造成零碎知识散落到各个角落，时间一长就变成“垃圾”了。
				- 对散落在各处的知识“孤岛”，它们通常会继续这样待下去，即使想每隔一段时间去整理也是头疼的很。
				- 所以，在构建个人知识库时，必须要限制：所有的记录必须在一个统一的地方去输入，Logseq 中的 Journal 无疑是最合适的位置。
			- 态 (State)：页面只用来存放整理好的知识 (MOC) 或自动查询的看板。
				- 有两种整理知识碎片的方向：自动查询 / 手动整理。
				- 自动查询
					- 对于 Project 这种类型的内容，通常是由很多 Task 组成，Logseq 中使用 Query 是最合适的。随着内容的增长，Project Query 会自动索引，省去手动整理的繁琐。
				- 手动整理
					- 这种内容一般属于 Area (长期关注、有价值的内容)。比如通过 Journal 碎片内容沉淀出来的方法论、编程中的某些概念或最佳实践等。
					- 这时候就不能使用 Query 自动查询了，自动查询的内容不具备结构化，而我们要沉淀出有长期价值的知识必然是结构化的知识体系。
			- 软链 (Soft-Link)：不要搞深层文件夹 (如 `Resource/Template/Daily`)，用 Tags `tags:: [[Resource]]` 来关联。
				- 有时候一个概念属于某一细分领域，比如 JavaScript 中的 IndexedDB，如果要创建一个 IndexedDB 页面作为相关知识点的 MOC，写成 `Area/Tech/JavaScript/IndexedDB` 会导致路径嵌套太深，后续很难维护和查阅。
				- 这时候一般将这一细分领域的内容直接挂到该领域上，比如 JavaScript/IndexedDB，而 JavaScript 由属于 Tech 领域。这里的 Tech 已经是一个概念非常宽泛的领域了，如果这是一个长期要关注的领域，再把 Tech 挂到 Area 领域中或者写成这样 `Area/Tech`。
				- 在将细分领域归类时，Logseq 中通常用 `Tags` 来建立“软连接”。
