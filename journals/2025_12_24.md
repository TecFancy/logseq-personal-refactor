- 09:05 [[Git]] Rebase 工作流与最佳实践 #reading
	- 核心概念：Rebase vs Merge
		- 目标：保持 Commit History 的整洁与线性 (Linear History)
		- 区别
			- |Merge|Rebase|
			  |--|--|
			  |保留真实时间线，但会产生分叉和合并节点 (Merge Commit)|“变基”，将当前分支的修改“重放” (Replay) 在目标分支的最新节点之后，就像从未分叉过一样|
	- 基础工作流：同步上游代码
		- 场景：开发 `feature/xxx` 分支时，`development` 分支有了更新
		- ✂️ Snippet: Pull with Rebase
			- `git checkout feature`
			- `git rebase development`
				- *更推荐习惯：`git pull --rebase origin development`*
		- 冲突处理
			- 修改文件 -> `git add .`
			- **关键点**：不要 commit，直接运行 `git rebase --continue`
			- 放弃：`git rebase --abort`
	- 进阶技巧：交互式变基 (Interactive Rebase)
		- 场景：提交 PR 前，清理本地琐碎的 “WIP” 或 “Fix typo” 等提交
		- 命令：`git rebase -i HEAD~n~` (n = 整理最近 n 个提交)
		- 指令速查
			- `pick`：保留提交
			- `squash` (s)：融合到上一个提交 (保留 Message)
			- `fixup` (f)：融合到上一个提交 (**丢弃 Message**，最常用)
	- Gotcha (黄金法则)
		- **绝对禁止**：在公共/共享分支 (如 `main`，`develop`) 上执行 Rebase
		- 原因：Rebase 会修改 Commit SHA-1，破坏历史。如果队友基于旧历史开发，会导致灾难性冲突
		- 范围：仅在**只有你一个人**的本地分支上操作
